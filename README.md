## Запуск проекта

Для запуска всех сервисов (backend, frontend, database) и выполнения тестов используйте команду в корневой директории проекта:

```bash
docker-compose up --build
```

Это запустит:
*   **Backend (Flask)**: Приложение будет доступно на `http://localhost:5000`.
    *   Документация API (Swagger UI) доступна по адресу: `http://localhost:5000/apidocs`.
*   **Frontend (React + Vite)**: Пользовательский интерфейс будет доступен на `http://localhost:3000`.
*   **База данных (PostgreSQL)**: Сервер базы данных будет доступен на хост-машине по порту `5433` (внутри Docker сети используется порт `5432`).
*   **Тесты (Pytest)**: Сервис `test` в `docker-compose.yml` автоматически запустит тесты при сборке и запуске. После выполнения тестов происходит реинициализация схемы БД, включая создание администратора по умолчанию и нескольких книг.

## Учетные данные администратора по умолчанию

После первого успешного запуска сервиса `test` (или ручного запуска скрипта `reinitialize_db_schema.py` в контейнере `backend`), будет создан пользователь-администратор со следующими учетными данными:
*   **Логин**: `admin`
*   **Пароль**: `adminpassword`

Этому пользователю также автоматически добавляется несколько книг для тестирования.

## Проверка функциональности

При проверке проекта обратите внимание на следующие аспекты:

### Backend (API)
*   **Аутентификация и Авторизация**:
    *   Реализована через Basic Authentication.
    *   Разделение прав доступа для обычных пользователей и администраторов.
*   **Модели данных**: `User`, `Book`, `Exchange`, `Review`.
*   **Основные эндпоинты (проверьте через Swagger UI на `http://localhost:5000/apidocs`)**:
    *   Регистрация (`/api/register`) и получение информации о себе (`/api/users/me`).
    *   CRUD операции для книг (`/api/books`). Владелец может редактировать/удалять свои книги. Удаление книги невозможно, если она участвовала в обмене; вместо этого ее можно пометить как недоступную.
    *   Просмотр списка доступных книг с фильтрацией по названию и автору.
    *   Управление обменами (`/api/exchanges`):
        *   Создание предложений обмена.
        *   Просмотр своих предложений (отправленных/полученных).
        *   Принятие, отклонение, отмена предложений.
        *   Принятие обмена автоматически меняет владельцев книг, делает их недоступными и отклоняет конфликтующие активные предложения по этим книгам.
    *   Управление отзывами (`/api/books/<book_id>/reviews`): Добавление и просмотр отзывов. Пользователь может оставить только один отзыв на книгу.
*   **Административные функции** (требуют прав администратора):
    *   Получение статистики по обменам (`/api/admin/exchanges/stats`).
    *   Просмотр списка всех книг в системе (`/api/admin/books`).
    *   Просмотр списка всех пользователей (`/api/admin/users`).

### Frontend (Пользовательский интерфейс)
*   **Взаимодействие с Backend API**: Фронтенд использует API бэкенда для всех операций. Прокси настроен в `frontend/vite.config.js` для перенаправления запросов с `/api` на `http://backend:5000`.
*   **Страницы и Навигация**:
    *   Главная страница, вход, регистрация.
    *   Список всех доступных книг с фильтрацией.
    *   Детальная страница книги с информацией, отзывами и формой предложения обмена (если применимо).
    *   Страницы для аутентифицированных пользователей: "Мои книги" (с возможностью редактирования), "Добавить книгу", "Мои обмены" (с возможностью управления).
    *   Страницы для администраторов: "Статистика обменов", "Все книги (Админ)", "Все пользователи (Админ)".
*   **Защищенные маршруты**: Проверьте, что доступ к защищенным страницам (для обычных пользователей и для администраторов) корректно работает.

### Тестирование
*   **Запуск**: Тесты автоматически запускаются сервисом `test` при выполнении `docker-compose up`.
*   **Покрытие**:
    *   Интеграционные тесты для API эндпоинтов находятся в `backend/tests/integration/`. Они проверяют основную логику API, включая аутентификацию, операции с книгами, обменами и административные функции.
    *   Базовые юнит-тесты для моделей находятся в `backend/tests/unit/`.
*   **Изоляция**: Тесты используют отдельную сессию базы данных и выполняют очистку таблиц до и после выполнения тестовых функций (см. `backend/tests/integration/conftest.py`), а также полную реинициализацию после всего набора тестов.
